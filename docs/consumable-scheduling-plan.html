Goal: turn the consumable fields you already capture into a live schedule that tells each responsible person what to order or replace, and by which date — so a coolant tank, oil filter, or welding wire never runs dry mid-production.
Starting point — fields already captured per consumable

Every consumable record today already stores the raw inputs a schedule needs. The gap is that nothing turns them into dates and actions.

stock on hand unit lead time (days) buffer (days) replacement schedule (days) last replaced (date) + burn rate (new)

The one missing input is consumption rate (how fast it's used up). We add it — entered directly, or better, derived from replacement history so it self-corrects.

Part A

The scheduling model

Two kinds of consumable, one set of dates the whole plant reads the same way.

A1

Two archetypes

Consumption-based — burns down continuously: coolant, lubricant, welding wire, shielding gas. Driven by a burn rate → reaches a reorder point.

Time / cycle-based — replaced on a cadence no matter what: oil change every 90 days, air filters, belts. Driven by last-replaced + frequency → a due date.

Many items are both. The engine computes both dates and warns on whichever comes first.

A2

The four dates it computes

Days of cover = stock ÷ burn rate
"At today's pace, N days of stock remain."
Reorder point = burn rate × (lead + buffer)
Order the moment stock drops to this level.
Order-by date = today + (cover − lead − buffer)
Last safe day to place the order.
Replace-due = last replaced + frequency
For time-based items; order-by = due − lead − buffer.

Part C

History makes it self-correcting

Every logged replacement sharpens the next prediction — the schedule learns your plant.

C1

Burn rate from actuals

Each time someone logs a replacement or refill, we store the event. From the interval between events and quantity used, we recompute the real burn rate.

burn = qty used ÷ days since last refill
Rolling average of the last 3–5 cycles, so it tracks reality, not the guess entered on day one.

If coolant was assumed to last 90 days but actually lasts 75, the order-by date quietly moves earlier — no one has to remember to fix it.

C2

Planned vs actual & production link

  • Planned vs actual per consumable — surfaces the ones you consistently mis-estimate.
  • Production-linked burn (optional): tie usage to units produced, so a busy month automatically pulls order dates in.
  • Supplier lead actuals: measure real order→receive time and feed it back into "lead" so the buffer isn't a guess.
  • Seasonality: flag items whose burn shifts with load, not just calendar.
Part D

From a date to an action

The schedule reaches out on WhatsApp and closes the loop — reusing the escalation engine you already built.

D1

Procurement loop

01 · SYSTEM
Order-by nears
Stock hits reorder point or due date approaches → a purchase suggestion is raised.
02 · WHATSAPP
Reminder sent
Responsible person is pinged: "Order 200 L coolant by 15 Jul." Ack / Snooze / Done.
03 · OWNER
Approve & order
Owner/Mnt Head approves; marking "ordered" starts the lead-time clock.
04 · SUPERVISOR
Mark received
Stock is topped up, last-replaced resets, the calendar recomputes forward.
05 · ESCALATE
If ignored
No ack before order-by? It escalates up — same timer used for breakdown tickets.
Reuse, don't rebuild: the ACK + escalation-timer infrastructure already shipped for breakdown tickets drives these reminders too — a due order that no one acknowledges escalates exactly like an unhandled machine fault.
