Diagnosis

What breaks the experience today

Both surfaces work — but they read as two separate apps bolted together.

01

Dashboard

wall of numbers
  • 12+ KPI cards rendered at the same size — "Machines Down" screams as loud as "Total Tickets." No visual triage.
  • The one thing that matters — what needs action now — sits below eight throughput metrics.
  • Owner-only ownership grid, custom-KPI config, and manual entry are stacked in one endless column.
  • Polls every 5s but numbers change with no motion or "updated" cue — you can't tell it's live.
02

Vault

long scroll
  • Machine management, supervisor management, anti-pilferage, and the machine picker are all stacked before you reach documents.
  • The machine picker — the primary context for everything below it — is buried mid-page.
  • Tabs (Documents / Parts / Consumables) live at the very bottom, easy to miss.
  • Feedback is a patchwork of inline green divs, alert(), and confirm().

Part A

UX point of view

One app shell, a clear hierarchy of attention, and a screen that greets each role by name.

Persistent left rail + top bar. Company, role, and a real "● live" indicator replace the duplicated userbars. One logout, one auth guard, one place for language.

A2

A screen tailored to each role

Same shell, different default view and emphasis — so the first screen answers each person's actual question.

Owner

"Is my plant healthy and is my money leaking?"

  • Lands on Overview: attention strip + cost/insight tiles
  • Supervisor ownership grid & risk map
  • Custom KPIs, team & machine control
Mnt Head

"What's broken and what do I fix next?"

  • Lands on Tickets / Needs Attention
  • Full Vault write access — docs, parts, consumables
  • MTBF / MTTR / repeat-breakdown insights
Supervisor

"My machines — anything down, any proof to log?"

  • Lands on My Machines (assigned only)
  • One-tap anti-pilferage: scan → photo → resolve
  • Mobile-first — most supervisors are on phones
A3

Dashboard: triage, not a wall

Rank the 12 metrics into four tiers so the eye lands on danger first:

  • Tier 1 — Needs action: Machines Down, Urgent, Stale. Large, color-coded, top of page.
  • Tier 2 — Throughput: Open, Closed today, Avg fix, Plant health. Compact row.
  • Tier 3 — Insight: MTBF, MTTR, Repeat %, #1 problem machine.
  • Tier 4 — Detail: weekly trend, recent tickets, custom KPIs.
Live cue: on each 5s poll, flash a subtle "updated · now" stamp and animate only the numbers that changed — so the screen visibly breathes.

A5

One consistent feedback & state system

Every action confirms

  • Toasts replace alert()
  • Inline validation on forms
  • Optimistic UI where safe

Every list has 4 states

  • Loading (skeleton, not "—")
  • Empty (guidance, not blank)
  • Error (retry, not dead)
  • Data

Destructive = deliberate

  • Modal confirm replaces confirm()
  • Names the target ("Delete Ramesh?")
  • Read-only roles see disabled, not hidden

B2

Behavior changes

  • Auth once: a single guard + token-refresh at the shell level, not duplicated in each page's script.
  • Pending approval (HTTP 403) gets a real screen — "Registration under review" — instead of an alert() + kick to login.
  • Polling with presence: keep the 5s poll & back-off, add a visible "last updated" + pause when tab hidden (already partly there).
  • Deep links: /app/machines/CNC-03/parts — every view addressable, back button works.
  • Manual KPI entry surfaces inline on the KPI it belongs to, with save confirmation.
B3

Technical foundation

  • Retire dangerouslySetInnerHTML — rebuild Dashboard & Vault as real React components with state & props.
  • Shared <AppShell> layout wrapping every authenticated route; role passed via context.
  • One design-token sheet (reuse the site's CSS variables) so Vault, Dashboard, and marketing site match.
  • Central API client: auth header, 401/403 handling, retry/back-off in one place.
  • i18n coverage extended into the app (currently English-only post-login).
